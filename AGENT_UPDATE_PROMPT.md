**目标**：从 `LLM_API_Providers_All.md` 中找出尚未被 README.md 收录的提供商，检索它们是否符合收录标准。

  2. 排除已在 README.md Part 1 或 Part 2 中出现的提供商

```

#### 2.2 对每个待检索提供商执行：

```
检索流程（每家至少 3 组搜索）：

  搜索 1（免费额度）：
    "{提供商名} free tier 免费 API"
    "{提供商名} 注册 赠送 token"

  搜索 2（编程套餐）：
    "{提供商名} coding plan 编程套餐"
    "{提供商名} developer subscription"

  搜索 3（最新动态）：
    "{提供商名} API {当前年份} 优惠 活动"

  如果搜索到有价值的信息 → 访问官方页面核实 → 提取字段
  如果搜索无结果 → 标记为"无公开免费额度/编程套餐"→ 不收录
```

#### 2.3 收录判定标准

```
【收录至 Part 1（白嫖指南）】满足任一：
  ✅ 有明确的免费注册赠送（Token / 代金券 / 积分 / 免费额度）
  ✅ 有永久免费层（Free Tier），无需付费即可持续使用
  ✅ 有正在进行的限时免费活动（未过期）
  ✅ 有实质性的学生/教育/初创优惠
  ✅ 有永久免费的模型可调用（0元模型）

【收录至 Part 2（编程订阅）】满足全部：
  ✅ 有明确的面向 AI 编程/Coding 场景的包月订阅
  ✅ 套餐名含 Coding / 编程 / 编码 / Programming / Developer 等关键词
  ✅ 固定月费 + 大量 API 调用额度
  ✅ 支持 Claude Code / Cursor / Cline 等主流编程工具

【不收录】：
  ❌ 完全无免费额度且无编程套餐
  ❌ 仅面向企业/需私下联系商务
  ❌ 平台已关闭或 API 服务已下线
```

### 阶段 3：重新排序 + 重算推荐 + 组装输出

#### 3.1 Part 1 重新排序

```
排序依据（按优先级）：
  1. 免费额度绝对价值（金额或Token数量，越大越前）
  2. 有效期长度（永久 > 1年 > 90天 > 30天）
  3. 使用便利性（无需实名 > 实名 > 企业认证）
  4. 模型质量和种类丰富度
  5. 额外福利（邀请奖励、学生优惠加分）

特别规则：
  - 永久免费层（如 Groq、Cerebras）的排序权重应高于大额但限时的赠送
  - 同等条件下，多模型平台排在单模型平台前面
```

#### 3.2 Part 2 重新排序

```
排序依据：
  1. 性价比（调用额度 / 月费）
  2. 入门门槛（首月优惠价 / 最低月费）
  3. 模型覆盖度（支持模型数量和质量）
  4. 工具兼容性（支持的IDE/Agent工具数量）
```

#### 3.3 重新生成横向对比表

  - 模型数量
  - 多模型切换

#### 3.4 重新生成综合推荐

重新评估并更新：

白嫖推荐 TOP 5：
  → 基于最新的免费额度数据，重新排出前5名

编程订阅推荐（按需求场景）：

  检索组 1："{提供商名} free tier 免费 API"
  检索组 2："{提供商名} coding plan 编程套餐"
  检索组 3："{提供商名} 注册 赠送 优惠 {当前年份}"

### 6.2 官方页面访问清单

```
必访页面（如果搜索发现变化或新信息）：
  1. 定价页（Pricing）
  2. 开发者文档首页

```
为提高效率，可按以下优先级安排核实顺序：

第一优先级（最可能发生变化）：
  - 有限时活动的条目（活动可能已过期或更新）
  - 有首月优惠的编程套餐（优惠可能已结束）
  - 近期有行业新闻的提供商（可能调价/新功能）

  - 注册赠送额度（额度可能上调/下调）

  - 前置条件（注册方式极少变化）
  - 推荐工具和致谢区域（不修改）

   可以只做一组快速搜索确认仍在运营即可，无需深入检索。

## 七、常见变化场景处理

### 7.1 价格调整

```
场景：提供商调整了 Coding Plan 价格

处理：
  1. 更新价格数字
  2. 如果涨价，在备注中标注"⚠️ {日期}起涨价"
  3. 更新横向对比表中的对应数据
  4. 重新评估排序和推荐位置
  5. 在变更日志中记录旧价格 → 新价格
```

### 7.2 活动过期

```
场景：某限时活动已过期

处理：
  - 如果提供商仍有其他免费额度 → 移除过期活动描述，保留其他信息
  - 如果过期后完全无免费额度 → 从 Part 1 移除该提供商
  - 在邀请/拉新奖励字段中，过期活动改为"未查到"或移除
  - 在变更日志中记录
```

### 7.3 新增提供商

```
场景：发现 LLM_API_Providers_All.md 中某家未收录的提供商有免费额度

处理：
  1. 按完整格式填写所有字段
  2. 按排序规则插入到 Part 1 或 Part 2 的正确位置
  3. 更新所有序号
  4. 更新横向对比表（如果是 Part 2 新增）
  5. 重新评估推荐排名
  6. 在变更日志中记录新增原因
```

### 7.4 提供商下线

```
场景：某平台关闭或停止 API 服务

处理：
  1. 从 Part 1 / Part 2 移除
  2. 更新后续序号
  3. 重新评估推荐排名
  4. 如果是 Part 2 提供商，更新横向对比表
  5. 在变更日志中记录
  6. 考虑在 LLM_API_Providers_All.md 中也标注状态
```

### 7.5 LLM_API_Providers_All.md 有新增提供商

```
场景：全量列表中新增了之前不存在的提供商

处理：
  1. 对新增提供商执行完整检索流程（阶段2的流程）
  2. 如果符合收录标准 → 按格式新增到 README.md
  3. 在变更日志中标注"来源：全量列表新增"
```

### 8.1 必须遵守的规则

```
铁律（违反即更新失败）：

  1. 【数据有据】每个数字必须有官方页面来源支撑，不可臆造
  2. 【时效标注】所有限时活动必须标注截止日期，过期必须清除
  3. 【完整核实】Part 1 和 Part 2 的每一个现有条目都必须核实，不可跳过
  4. 【格式一致】输出格式必须与现有 README.md 完全一致
  5. 【日期更新】免责声明中的日期必须更新为当前执行日期
  6. 【变更记录】必须输出变更日志文件
  7. 【不擅改结构】头部介绍区、推荐工具区、致谢区不可修改
```

### 8.2 容易出错的点

```
⚠️ 常见陷阱：

  1. 【代金券≠Token】代金券是钱可抵扣费用，Token是调用量，单位不同
  2. 【单模型额度×模型数=总额度】有的平台是"每模型送X"，总量需乘以模型数
  3. 【Coding Plan次数≠模型调用次数】一次用户提问≈5-30次模型调用
  4. 【过期优惠当新优惠】必须核实活动截止日期，过去的优惠不能展示
  5. 【混淆产品线】云服务代金券≠大模型API额度，不要混淆
  6. 【遗漏限制条件】免费层的RPM/RPD限制必须注明
  7. 【URL失效】如果官方URL有变动，必须更新链接
```

  - 利用并行搜索能力（如可用）
```

## 三、核心工作流——逐平台独立检索

> **核心机制**：每个平台独立执行完整的「搜索→访问→提取→存储」闭环，平台之间互不干扰。

### 3.1 执行顺序

```
1. 先逐一核实 README Part 1 现有条目（标记：无变化 / 需更新 / 已失效）
2. 再逐一扫荡「待扫荡列表」中的每家提供商
3. 所有平台检索完毕后，按检索完成顺序合并产出最终文档
```

### 3.2 单平台检索流水线

对**每一个**平台，独立执行以下完整流程：

```
步骤 1：多层搜索（见 3.3 节搜索策略）
步骤 2：访问搜索结果中的官方页面，获取并阅读完整页面内容
步骤 3：提取字段信息（见 3.5 节字段模板）
步骤 4：判定是否符合收录标准（见 3.6 节）
         → 符合：生成 {平台名}.md 文件到当次检索目录
         → 不符合：不生成文件，仅在 _progress.md 中记录结果
步骤 5：更新 _progress.md，标记该平台检索完成，进入下一平台
```

> ⚠️ **强制要求**：
> - 必须实际访问页面并获取完整内容核实，**严禁仅凭搜索摘要填写信息**
> - 每条关键信息（额度数字、有效期、活动规则）必须标注来源 URL
> - 如页面无法访问，记录失败原因并在备注中说明
> - **不符合收录标准的平台不生成 markdown 文件**，避免合并时干扰

### 3.3 三层递进式搜索策略

对每个平台，按以下三层递进执行搜索。`{name}` 包含该平台的**所有已知名称变体**（中文名、英文名、缩写、产品线别名）。

#### 第一层：核心搜索（必做）

```text
中文：
  "{name} 免费额度 API {当前年份}"
  "{name} 注册 赠送 token 代金券 {当前年份}"
  "{name} 邀请奖励 推荐计划"

英文（国外平台必做）：
  "{name} free tier API pricing {当前年份}"
  "{name} free credits sign up new user"
```

#### 第二层：扩展搜索（必做）

```text
  "{name} 新用户 福利 优惠 活动 {当前年月}"
  "{name} 学生 教育 优惠"
  "{name} free model 0元 永久免费"
  "{name} changelog 公告 额度调整 {当前年份}"
  "{name} API 评测 免费 {当前年份}"  ← 第三方评测/社区讨论
```

英文补充（国外平台）：
```text
  "{name} student academic discount developer program"
  "{name} free tier changes announcement {当前年份}"
```

#### 第三层：深度搜索（前两层未获有效信息时执行）

```text
  site:zhihu.com "{name} 免费"
  site:reddit.com "{name} free tier"
  site:x.com "{name} free credits"
  "{name} 开发者 博客 免费额度"
  "{name} GitHub discussion free"
```

> **搜索原则**：
> - 中英文搜索**必须同时执行**
> - 加入时间限制（优先近 6 个月内容）
> - 平台名称需覆盖所有变体（如「硅基流动」+「SiliconFlow」+「siliconflow」）

### 3.4 必访页面

搜索后，对有价值结果**必须访问以下页面核实**（每家至少前 3 个）：

1. 官方定价页（`/pricing` 或 `/billing`）
2. 开发者文档首页（Getting Started）
3. 活动/促销页面（首页 Banner、活动中心）
4. API 控制台/Dashboard（免费 Tier 说明）
5. 官方博客/公告（近 3 个月）

### 3.5 字段提取模板

对每个平台，按以下字段提取（查不到填「未查到」）：

| 字段 | 要求 |
|------|------|
| **平台名称** | 官方中英文名称 + 平台 URL |
| **注册福利** | 具体赠送内容（多少 Token / 多少元 / 什么免费模型），核心数字**加粗** |
| **有效期** | 永久 / X天 / X个月 / 限时（写明截止日期） |
| **邀请/拉新奖励** | **必须写明具体情况**（见下方详细要求），严禁模糊描述 |
| **学生福利** | 有无教育/学生优惠，具体内容 |
| **前置条件** | 注册要求（邮箱/手机/实名/信用卡等） |
| **备注** | 支持模型、特色功能、限速信息等 |

#### ⚠️ 邀请/拉新奖励字段的详写要求

**严禁**写「有邀请活动」「邀请新用户获额外福利」等模糊描述。必须写清以下全部要素：

| 要素 | 说明 |
|------|------|
| 活动名称 | 如「推荐官」计划、「邀请有礼」等 |
| 奖励内容 | 具体数字：双方各得多少 Token/代金券/积分 |
| 叠加规则 | 是否可无限叠加、有无上限 |
| 有效期 | 长期有效 / 截止日期 |
| 参与方式 | 邀请好友完成什么操作可触发奖励 |

**参考范例**（README 中硅基流动的写法）：
> **「推荐官」计划**（长期有效）：每成功邀请1位好友完成实名注册，双方均可获赠约14元通用代金券（等值2000万Tokens），**奖励可无限叠加，上不封顶**。

### 3.6 收录判定

满足**任一**条件即收录至 Part 1：
- ✅ 有明确的免费注册赠送（Token / 代金券 / 积分）
- ✅ 有永久免费层（Free Tier），无需付费可持续使用
- ✅ 有正在进行的限时免费活动（未过期）
- ✅ 有实质性的学生/教育/初创企业优惠
- ✅ 有永久免费模型可调用（0 元模型）

**不收录**：仅「14天试用」（除非额度极大）/ 纯企业 ToB / 完全无免费额度 / 平台已关闭。

### 3.7 检索缓存目录与断点续执

#### 目录结构

每次全量检索使用**独立的日期目录**，历史检索永久保留，互不覆盖：

```
项目根目录/
├── _检索缓存/
│   └── 免费资源/
│       ├── 2026-06-19/              ← 本次检索
│       │   ├── _progress.md          ← 检索进度记录
│       │   ├── 硅基流动.md            ← 合格平台（仅合格才有文件）
│       │   ├── Groq.md
│       │   └── ...
│       ├── 2026-07-08/              ← 下次全量检索
│       │   └── ...
│       └── ...                       ← 历史检索永久保留
```

#### 进度记录 `_progress.md`

```markdown
# 检索进度记录

> 执行日期：{YYYY-MM-DD}
> 基准文件：LLM_API_Providers_All.md（共 {N} 家）
> 执行模式：全量执行 / 断点续执

## 已完成

| # | 平台名称 | 结果 | 完成时间 |
|---|---------|------|----------|
| 1 | 硅基流动 | ✅ 合格（已生成文件） | 2026-06-19 10:30 |
| 2 | xxx平台 | ❌ 不合格（无免费额度） | 2026-06-19 10:35 |

## 待执行

| # | 平台名称 | 状态 |
|---|---------|------|
| 3 | yyy平台 | ⏳ 待检索 |
```

#### 执行模式

| 模式 | 触发条件 | 行为 |
|------|---------|------|
| **全量执行** | 当天日期目录不存在 | 创建 `_检索缓存/免费资源/{当天日期}/`，生成 `_progress.md`，从头检索 |
| **断点续执** | 当天日期目录已存在且 `_progress.md` 中有「待执行」条目 | 跳过已完成的平台，从待执行的第一个继续 |

> - 想**强制全量重新检索** → 删除当天日期目录或使用新日期
> - 想**接着中断继续** → 直接执行，自动识别当天目录并续检
> - 历史目录**只读保留**，不修改不删除

#### 合并产出

```
全部检索完成后：
  1. 从当次日期目录读取所有 .md 文件（排除 _progress.md）
  2. 目录中只有合格平台的文件，无需过滤
  3. 按检索顺序编排序号（1, 2, 3...），组装为最终输出
  4. 排序由后续 Agent 负责，本 Agent 无需参与排序逻辑
```

## 五、质量铁律

1. **数据有据**：每个数字必须有官方页面来源，不可臆造
2. **时效标注**：限时活动必须标注截止日期，过期必须清除
3. **完整核实**：Part 1 每个现有条目都必须核实，不可跳过
4. **全量扫荡**：全量名单中每家未收录提供商都必须检索
5. **拉新详写**：邀请/拉新活动必须写清具体规则（见 3.5 节），严禁模糊描述
6. **格式一致**：输出格式与现有 README 保持一致
7. **日期更新**：免责声明中的日期更新为当前执行日期
8. **不擅改结构**：头部介绍区、Part 2、推荐工具区、致谢区、推荐排名区均不可修改
9. **独立检索**：每个平台独立执行完整检索流水线，不得跨平台混淆信息

### 易错提醒

| 陷阱 | 说明 |
|------|------|
| 代金券 ≠ Token | 代金券是钱（可抵扣费用），Token 是调用量，单位不同 |
| 单模型额度 × 模型数 | 有的平台「每模型送 X」，总量需乘以模型数 |
| 过期优惠当新优惠 | 必须核实活动截止日期 |
| 混淆产品线 | 云服务代金券 ≠ 大模型 API 额度 |
| 遗漏限制条件 | 免费层的 RPM/RPD 限制必须注明 |
| URL 失效 | 官方 URL 变动必须更新 |
